## 4. 핵심 요구사항 (사용자 원본 요구 12항)

1. **실행 모드 2종**
   - (a) **vllm-mlx 모드** — MLX 기반 / `mlx-community` 모델 전용
   - (b) **CPU 모드** — 비-MLX 모델 또는 CPU 성능 확인용 (`llama.cpp` + `transformers` 둘 다 지원)

2. **측정 항목** — TPS(tokens/sec), 컨텍스트 크기별 실측 RAM 사용량, CPU 사용률. 전부 실측만, 추정치 금지.

3. **채팅으로 직접 성능 테스트** — 일반 채팅, 코드 작성, 이미지 분석.

4. **컨텍스트 크기 세밀 조정 + 쉬운 모델 전환**.

5. **모델 종류별 전용 테스트 환경** — 멀티모달, 이미지 전용, 이미지 생성, 음성(TTS/ASR), LLM.

6. **직관적 UI** — 누가 봐도 기능별 구성·사용이 쉬워야 함.

7. **아키텍처** — Rust 코어가 메인, Python은 래핑만 (`python3.12` + `uv`), Flutter macOS 데스크탑 앱.

8. **Rust 실시간 자원 모니터링 + 고속 자동 제어** — 메모리 한계 접근 시 즉시 모델 언로드 (테스트용 앱이라 급정지 허용).

9. **전체 라이프사이클 1사이클** — 버튼 → 서버 시작(python subprocess) → 테스트 → 정지 버튼 → 모델 언로드 + 정리 + 서버 종료.

10. **추가 직관적 테스트·시각화 기능 적극 포함** (시각화 우대) — 5절의 번들 전부 확정.

11. **간단한 내부 DB(SQLite)** — 모델명 + 실측 데이터 저장 → 모델 간 차트 비교 용이.

12. **모델별 차이를 예외 처리로 덮지 않음** — Qwen 하이브리드 캐싱 vs llama 차이 등. 실작동 + 최속 실시간 제어로 해결.

## 6. 설계 전제 6개 (P1~P6, 사용자 전부 승인)

### P1. v1 올인 범위

4개 번들 전부 + 필요한 것 추가. 긴 개발 기간 허용.

### P2. 실측 only

- RAM은 `phys_footprint` + Metal `currentAllocatedSize` 컴포지트로 실측
- 업계는 추정식(weights + KV cache 공식) 사용 → 실측이 핵심 차별점

### P3. Rust 자원 감시

- 실시간 모니터링
- 메모리 한계 임박 시 즉시 모델 언로드
- 급정지 허용

### P4. 백엔드 추상화

- `vllm-mlx` 메인 + `mlx-lm` 폴백
- 백엔드를 어댑터 계층 뒤에 둠
- MLX 생태계가 빠르게 변해서(Ollama MLX 프리뷰 등) 추후 백엔드 교체가 어댑터 하나 추가로 가능해야 함

### P5. 모델 프로파일 시스템 (모델별 기대형식 다양성 해법)

| 항목 | 내용 |
|---|---|
| 프로파일 단위 | 모델마다 config-driven 프로파일(JSON 파일) |
| 서버 모드 정규화 | OpenAI 호환 API (chat template은 서빙 서버가 처리 — 업계 검증된 방식) |
| 프로파일 선언 내용 | 모델 종류(LLM/멀티모달/이미지생성/TTS/ASR), 기대 입출력 형식(chat/JSON/image/audio), 기본 파라미터 |
| 초안 자동 생성 | HF `config.json` 자동 파싱으로 프로파일 초안을 자동 생성하고, 사용자가 수정·추가 가능 |
| 확장 방식 | 새 형식의 모델이 나와도 프로파일 파일 하나 추가로 대응 — 코드 수정 불필요 |

### P6. 문서화 기반 확장성

- Docs에 기능별 구현 문서 + 확장 포인트 명세를 둔다
- 추후 기능 추가, AI 형식 변경·신규 추가·맞춤 변경 시 문서만 보고 작업 가능해야 한다

## 7. 아키텍처 접근 방식 (확정: 코어 우선 2단계)

### 1단계: Rust 코어 + CLI

- Rust 코어 + CLI를 먼저 완성·검증
- 실측 측정, 라이프사이클 제어, 메모리 와치독 등 가장 위험한 부분을 UI 없이 먼저 증명
- 모든 기능이 CLI로 실행 가능해야 함 (자동화 테스트·재현 검증 용이)

### 2단계: Flutter UI 탑재

- 검증된 코어 위에 Flutter UI 탑재

### 선택 이유

- 측정 엔진 설계 결함이 늦게 발견되면 UI까지 연쇄 재작업 → 위험을 앞단계에 몰아 검증
- Grok 작업 단위(경계 명확)에도 유리

### 기각된 대안

| 대안 | 기각 사유 |
|---|---|
| (a) 일체형 동시 개발 | 재작업 위험 큼 |
| (b) Python 중심 경량형 | "Rust가 메인 코어" 요구와 충돌 |
